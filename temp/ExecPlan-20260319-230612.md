# ExecPlan: Execute Simulator Rebuild From the Approved Plan

## Purpose

Execute the simulator and gem5 integration rebuild described in:

- `temp/plan-reimpl-sim-0319-0.md`

This ExecPlan is an implementation tracker. It does not redefine simulator
semantics. If an implementation detail conflicts with the approved rebuild
plan, the approved rebuild plan wins.

## Design Authority

Priority order for behavioral decisions:

1. human-authored `temp/draft-*.md`
2. human-authored `temp/design-*.md`
3. `temp/plan-reimpl-sim-0319-0.md`
4. FCC `docs/spec-*.md`
5. Loom `docs/spec-*.md`
6. explicit new design decisions made during implementation and documented back
   into the rebuild plan or specs

`temp/plan-*.md` and `temp/ExecPlan-*.md` other than the approved rebuild plan
are execution artifacts, not higher-priority design sources.

## Scope

Implement a shared cycle-accurate simulation kernel that:

- simulates mapped execution on ADG hardware topology
- is reusable from standalone and gem5
- replaces the current replay-bridge path as the primary gem5 architecture
- emits JSON trace usable by visualization playback

## Non-Scope During Initial Execution

- rewriting unrelated mapper subsystems
- preserving the old functional backend as a timing model
- direct accelerator-to-accelerator streaming links in gem5

## Acceptance Gates

The implementation is complete only when all of the following are true:

1. `ninja -C build check-fcc-unit` passes
2. standalone mapped simulation passes for:
   - `sum-array.sum-array-demo-chess-6x6`
   - `vecadd.vecadd-demo-chess-6x6`
3. standalone emits versioned JSON trace
4. visualization playback works for the two mapped cases above with:
   - `Simulation: On/Off`
   - `Stop`
   - `Step`
   - `Back`
   - `Auto Play`
5. gem5 primary path no longer depends on replay bridge
6. gem5 smoke passes for `sum-array.sum-array-demo-chess-6x6`
7. updated specs describe the new architecture and trace playback path

## Execution Strategy

The implementation follows strict milestone gating:

- do not start a later milestone before the current milestone acceptance checks
  pass
- keep replay bridge available only as migration fallback until the embedded
  path is stable
- prefer replacing behavior through one shared kernel rather than duplicating
  logic between standalone and gem5

### Milestone 1: Static Model and Config Decode

Goals:

- build the ADG topology loader
- build the config bitstream decoder
- build the DFG mapping overlay model
- extract software-visible completion obligations from the overlay

Acceptance:

- topology loader enumerates modules and channels correctly
- config decode round-trip tests pass against `ConfigGen`

Current implementation status:

- `StaticMappedModel` added under `include/fcc/Simulator/StaticModel.h` and
  `lib/fcc/Simulator/StaticModel.cpp`
- the model currently extracts:
  - flattened hardware modules
  - hardware channels
  - flattened hardware ports with value/tag width metadata
  - PE container metadata from `ADGFlattener::getPEContainment()`
  - boundary input and output bindings
  - memory bindings
  - completion obligations
- channel descriptors now also preserve `pe_input_index` and
  `pe_output_index` edge attributes for later PE-container simulation
- `SimSession::loadConfig(...)` now accepts `ConfigGen::ConfigSlice`
  information in addition to raw `config.bin` bytes
- `SimSession::buildFromMappedState(...)` now has an overload that accepts
  `PEContainment` records, and `tools/fcc/fcc.cpp` passes
  `flattener.getPEContainment()` into the simulator session
- `tools/fcc/fcc.cpp` standalone simulation path now passes
  `configGen.getConfigSlices()` into the simulator session
- `CycleSimulationBackend` is now the default `SimSession` backend and builds
  the new static model while still delegating execution to the legacy
  functional backend
- `CycleKernel` now owns:
  - logical per-port signal state
  - source/destination connectivity derived from ADG edges
  - primitive-module instances for:
    - boundary sentinels
    - `add_tag`, `map_tag`, `del_tag`
    - `fifo`
    - `spatial_sw`
    - `temporal_sw`
  - fixed-point evaluation with `max_comb_iterations = 4`
  - versioned trace emission
- `CycleSimulationBackend` can now execute primitive-only mapped models on the
  new kernel instead of unconditionally falling back to the legacy backend
- `sum-array` e2e and mapper-only `vecadd` continue to pass after these
  simulator-layer changes
- memory-bridge mapper regressions triggered during the rebuild were repaired
  to keep the standing mapper gates healthy:
  - bridge-aware direct memory port binding now fails explicitly on type and
    resource mismatches
  - tagged egress width positive cases route again
  - tagged collide negative cases now fail earlier at placement/bridge binding
    and the unit expectations have been aligned to that earlier failure mode
- the new kernel is still incomplete for PE and memory semantics, so
  `FunctionUnit`, `memory`, and `extmemory` remain on the legacy backend path
  for now
- simulator runtime-image artifacts now include a dedicated control section:
  - `start_token_port`
  - scalar slot bindings
  - memory slot bindings
  - output slot bindings
- runtime image schemas were version-bumped to carry the control image
- the shared runtime subset has been further decoupled from builder-only
  dependencies:
  - `StaticMappedModel` is now a pure container type
  - `buildStaticMappedModel(...)` moved into a separate builder layer
  - `CycleKernel.cpp` no longer uses `llvm::ArrayRef`
  - this keeps the runtime subset closer to gem5-compilable form
- while debugging `vecadd`, one root cause was confirmed in the frontend:
  - `scf.for` body-local memory control had been wired to the raw
    `dataflow.stream` condition instead of the gated `dataflow.gate`
    condition
  - FCC now records both `forRawConds` and `forBodyConds`
  - loop-boundary iter-arg carries continue to use the raw `N+1` stream
  - memory-control builder now uses the body-visible `N` stream
  - regenerated `vecadd.dfg.mlir` now shows load/store control carries and
    their `cond_br` splits driven by `%afterCond`, not `%willContinue`

### Milestone 4: PE and FU Timing

Goals:

- implement `spatial_pe`
- implement `temporal_pe`
- implement ordinary FU timing
- implement dedicated `dataflow.*` FU state machines
- preserve the rule that `dataflow.*`-containing FUs are not legal inside
  `fabric.temporal_pe`

Acceptance:

- latency/interval tests pass
- temporal PE arbitration and busy-behavior tests pass

Current implementation status:

- simulator-side `function_unit` execution support added under:
  - `include/fcc/Simulator/SimFunctionUnit.h`
  - `lib/fcc/Simulator/SimFunctionUnit.cpp`
- the new kernel can now execute supported `fabric.function_unit` nodes inside
  `spatial_pe` when:
  - the FU body is a single supported op
  - the model does not require `memory` or `extmemory`
  - the FU does not belong to `temporal_pe`
- supported FU families currently include:
  - ordinary arithmetic and compare ops
  - `handshake.constant`
  - `arith.select`
  - `handshake.mux`
  - `handshake.join`
  - `handshake.cond_br`
  - dedicated `dataflow.stream`, `dataflow.carry`, `dataflow.invariant`,
    `dataflow.gate`
- `ADGFlattener` now preserves both `latency` and `interval` on flattened FU
  nodes for simulator-side decode
- `CycleSimulationBackend` no longer rejects every `FunctionUnit` node; it now
  enables the new kernel for supported spatial-FU-only models and still falls
  back for:
  - `temporal_pe`
  - no longer falls back for `handshake.load`, `handshake.store`, `memory`,
    and `extmemory`; those now execute on the new kernel
  - still falls back for `temporal_pe`
- `CycleKernel` no longer uses direct source-port mirroring as the main token
  transport model; it now keeps one buffered `SimChannel` per mapped hardware
  edge and commits transfers through `edgeState_`
- `SimMemory` now uses globally composed token generations for responses and no
  longer double-increments output generations when placing a completion token
- current status:
  - mapper-only `sum-array` and `vecadd` remain good
  - runtime replay now builds the simulator directly from
    `RuntimeImage.staticModel` rather than reconstructing an incomplete
    `MappingState` overlay
  - `sum-array` runtime replay now completes successfully on the new kernel
    with the correct output value `36`
  - `vecadd` runtime replay now also completes successfully on the new kernel
  - the deadlock that remained after the value bug was fixed turned out to be
    tail-edge residue that no longer blocks completion once obligations are
    satisfied and no module or memory work remains

Goals:

- prove `DmaDevice` viability
- validate `getPort()` and DMA callback wiring

Acceptance:

- isolated gem5 DMA spike works as expected

Current implementation status:

- `FccDmaSpikeDevice` landed and remains the focused proof that FCC's gem5
  extras can inherit `DmaDevice`, expose an MMIO aperture, and complete DMA
  callbacks through gem5 events
- `FccCgraDevice` direct-runtime path now:
  - loads `<case>.simimage.bin`
  - builds `CycleKernel` in-process
  - emits reply artifacts compatible with the existing gem5 runner contract
- `FccCgraDevice` has now been migrated in source from `BasicPioDevice` to
  `DmaDevice`
- `configs/fcc_cgra.py` now connects the accelerator `dma` port to
  `system.membus.cpu_side_ports`
- direct invocation is now being converted away from synchronous
  `sys->physProxy` access toward event-driven DMA staging:
  - memory regions are staged into local buffers via DMA read callbacks
  - kernel execution is launched from a scheduled event
  - post-execution memory images are written back through DMA write callbacks
- current blocker:
  - the updated gem5 device path still needs a clean gem5 build-and-smoke pass
    after the `DmaDevice` migration
  - per-request memory boundary integration has not yet replaced the current
    invocation-scope DMA staging model
